# CLAUDE.md

Guidance for Claude / AI agents working on **u1d-volume-dashboard**.

Read this file before making changes. It captures the invariants that are not obvious from the code and the rules that must hold for board-grade reporting.

---

## What this project is

Monthly operations dashboard for **U1Dynamics Manufacturing LLC**. It powers the operations report presented to the board of directors. Two data domains live in the `u1d_ops` Postgres schema:

- **Volume** — billed gallons per (period × customer × package). 32 validated months loaded as of Phase 1 (Jan 2023 → Mar 2026).
- **Production** — daily gallons/pallets per production line, joined to installed and 80%-target capacity. 1,595 daily rows across 9 lines as of Phase 1.6.

The dashboard runs in English. The board deck (`.pptx`) is generated by a separate workflow and stays in Spanish.

## Storage — Railway Volume (Phase 1.7)

**Decision:** Phase 1.7 uses Railway Volume for original Excel file storage. No S3, no R2, no SharePoint. This is intentional — keep the dependency surface small until volume justifies blob storage.

### Mount

The Railway Volume is mounted at `/app/storage` in production.

### Runtime root resolution

All file I/O must resolve the storage root through this exact precedence:

```ts
const STORAGE_ROOT =
  process.env.U1D_FILE_STORAGE_ROOT ||
  process.env.RAILWAY_VOLUME_MOUNT_PATH ||
  "/tmp/u1d-storage";
```

- `U1D_FILE_STORAGE_ROOT` — explicit override for local dev or migration.
- `RAILWAY_VOLUME_MOUNT_PATH` — set automatically by Railway when a volume is attached.
- `/tmp/u1d-storage` — last-resort dev fallback; never used in production.

### Write-time discipline

- **Do not write files during build time.** No `next build` step touches the filesystem under `STORAGE_ROOT`.
- **Do not write files during pre-deploy or migrations.** `db:migrate` and `db:seed:*` must not create directories under `STORAGE_ROOT`.
- **All file writes happen at runtime through the admin upload flow.**

A startup check (`ensureStorageRoot()`) runs at first use of the upload route. It confirms `STORAGE_ROOT` exists and is writable; if missing, it creates the directory tree at runtime. The check never runs from a migration or seed script.

### File layout

```
{STORAGE_ROOT}/u1d-volume-files/{year}/{month}/v{version_no}/{file_name}
```

Example:

```
/app/storage/u1d-volume-files/2026/03/v1/U1DYNAMICS_VOLUME_2026_03__a1b2c3d4.xlsx
```

### File naming convention

```
U1DYNAMICS_VOLUME_{YYYY}_{MM}__{sha256_prefix}.xlsx
```

- `{YYYY}` — four-digit year, zero-padded.
- `{MM}` — two-digit month, zero-padded.
- `{sha256_prefix}` — first 8 hex chars of the SHA-256 of the file buffer. Embedding it in the filename gives a self-describing artifact for forensic checks.

### Internal URL convention (stored in `original_blob_url`)

```
railway-volume://u1d-volume-files/{year}/{month}/v{version_no}/{file_name}
```

The `railway-volume://` scheme is internal. Any code that resolves this URL must go through the storage root resolver above. **Never hardcode `/app/storage` in code paths.** This keeps a future migration to S3/R2 a one-resolver change.

## Storage rules (Phase 1.7 invariants)

These ten rules are the contract. Any code change that breaks them is a regression.

1. The uploaded original Excel must be written to Railway Volume before parsing is finalized. Write first, parse second, DB commit third.
2. Compute SHA-256 from the file buffer (not from a temp file path).
3. Reject exact duplicate hashes (return 409, do not duplicate in storage or DB).
4. If the same period receives a different hash, create the next `version_no` for that period.
5. Only one active file version may exist per (period_year, period_month).
6. Superseded versions must remain in history (`is_superseded = TRUE`, `superseded_by_file_id` set).
7. Never delete prior uploaded files automatically. Manual delete only via admin tool, with audit log.
8. Board reports must read from locked Postgres data, never re-parse the Excel at report time.
9. Admin review pages may link to the stored original file for audit and download.
10. A startup check confirms the storage root exists and is writable; if missing, create it at runtime (admin-route boot path only, never in migrations or seed).

### Orphan-file recovery (rule 11)

If `persistOriginal()` writes the file to Railway Volume but the subsequent DB transaction fails (network blip, constraint violation, deploy mid-flight), the file remains on disk while no `volume_files` row references it. This is an orphaned runtime artifact.

On the next upload retry the route may encounter the orphaned path. It must:

1. Compute the SHA-256 of the inbound buffer (rule 2).
2. Probe the DB: is there a `volume_files` row whose `file_hash` matches?
   - **Yes** → return 409 (rule 3). This is a true duplicate retry; the prior upload actually committed.
3. Probe the disk: does the target path already exist?
   - **No** → normal path, call `persistOriginal()`.
   - **Yes** → it is an orphan. Read the file from disk, compute its SHA-256.
     - **Disk hash matches inbound hash** → reuse the existing file. Skip `persistOriginal()` and proceed to the DB transaction with the existing `filePath` / `blobUrl`.
     - **Disk hash differs** → allocate the next filesystem-safe `version_no`: take `MAX(version_no)` from `volume_files` for that period, walk the disk to find the next unused `v{N}` directory, pick `N+1` of the higher of the two. Never silently overwrite.
4. The retry logic runs inside the same SELECT current active file `FOR UPDATE` transaction window as a normal upload (PR-1.7 transaction order, step 1) so concurrent retries serialize correctly.

This logic is the upload route's responsibility, not the storage helpers'. The helpers refuse to overwrite via `flag: "wx"` (rule 7); the route decides whether to reuse, version-up, or surface the collision.

## Database — Phase 1.7 extensions to `u1d_ops.volume_files`

Migration `005_create_close_workflow.sql` adds the close-workflow schema. The volume file registry gains:

| Column | Type | Purpose |
|---|---|---|
| `version_no` | `INT NOT NULL DEFAULT 1` | Per-period version sequence. v1 is the first upload. |
| `is_active` | `BOOLEAN NOT NULL DEFAULT TRUE` | Exactly one row per period with `is_active = TRUE`. |
| `is_superseded` | `BOOLEAN NOT NULL DEFAULT FALSE` | Mirror of `is_active = FALSE` for legibility in queries. |
| `superseded_by_file_id` | `BIGINT REFERENCES u1d_ops.volume_files(file_id)` | Points to the version that replaced this one. |
| `original_file_path` | `TEXT` | Absolute path on the storage volume, relative to `STORAGE_ROOT`. |
| `original_blob_url` | `TEXT` | Internal `railway-volume://...` URL. |
| `storage_provider` | `TEXT NOT NULL DEFAULT 'railway-volume'` | `'railway-volume'` for Phase 1.7. Future values: `'s3'`, `'r2'`. |
| `uploaded_by` | `TEXT NOT NULL` | Email of the admin who uploaded (from NextAuth session). |
| `uploaded_at` | `TIMESTAMPTZ NOT NULL DEFAULT NOW()` | Replaces or complements `ingested_at`. |
| `staged_at` | `TIMESTAMPTZ` | Set when parsing succeeds and the file enters `in_review`. |
| `reviewed_at` | `TIMESTAMPTZ` | Set when an admin acknowledges all alerts. |
| `locked_at` | `TIMESTAMPTZ` | Set when the period is locked. Board reports gate on this. |

The pre-Phase-1.7 `UNIQUE (period_year, period_month)` constraint is **replaced** by `UNIQUE (period_year, period_month, version_no)` plus a partial unique index `UNIQUE (period_year, period_month) WHERE is_active = TRUE` to enforce rule 5.

A `board_periods` table tracks period lifecycle with `status` (`open / staged / in_review / locked / superseded / reopened`), `active_file_id` pointing at the active version, and audit timestamps (`reviewed_at` / `locked_at` / `reopened_at` plus corresponding `*_by` fields).

`package_alerts`, `customer_alerts`, and `data_quality_alerts` tables capture unknown labels and integrity flags seen during parse; a period cannot transition to `locked` while any alert is in `pending` status.

`monthly_operator_notes` stores the Markdown commentary for the deck's operator-narrative slides (capacity, supply chain, quality, initiatives, risks).

## Workflow lifecycle

```
upload .xlsx
   -> write to Railway Volume (rule 1)
   -> SHA-256 hash (rule 2)
   -> reject if duplicate hash (rule 3)
   -> parse with volume-parser.ts
   -> resolve version_no for the period (rule 4)
   -> INSERT volume_files (rules 5, 6)
   -> INSERT volume_fact rows
   -> INSERT alerts for unknown labels
   -> board_periods.status = 'in_review', active_file_id = new_file_id

admin review
   -> resolve alerts (map raw label -> catalog or ignore)
   -> acknowledge totals + reconciliation
   -> operator notes complete
   -> board_periods.status = 'locked', locked_at / locked_by set
   -> mv_volume_reconciliation refreshed

board report (Phase 2+)
   -> reads only from locked periods (rule 8)
   -> never re-parses Excel
   -> may link to stored original for audit (rule 9)
```

## What NOT to do

- **Do not** implement S3, R2, Cloudflare blob store, or any external object storage in Phase 1.7. Railway Volume is the only storage backend.
- **Do not** implement SharePoint pickup in Phase 1.7. That is Phase 3.
- **Do not** implement the `.pptx` deck generator in Phase 1.7. That is Phase 2.
- **Do not** read the original Excel at board-report time. Reports must source from Postgres.
- **Do not** hardcode `/app/storage` anywhere. Use the resolver.
- **Do not** write files from `npm run build`, `db:migrate`, `db:seed:volume`, or `db:seed:production`.
- **Do not** delete files from storage on supersede. Mark as superseded; keep the artifact.
- **Do not** trust the source TOTAL row from the Excel. Use the per-customer reconstruction (the parser already does this; the FY2024 TERRA-450 cases are the canonical reason).
- **Do not** add new packages or customers silently. Route them through `package_alerts` / `customer_alerts`.

## Common tasks for an AI agent

| Task | Where to look | Watch out for |
|---|---|---|
| Add a new package | `db/migrations/002_seed_catalogs.sql` + `volume-parser.ts` `PACKAGE_KEYS` | Must update both. Container_type and family enums are constrained. |
| Add a new customer | `002_seed_catalogs.sql` + `volume-parser.ts` `CUSTOMER_KEYS` + canonicalization | Parser uses `startsWith` — fragile. Add an alias map. |
| Touch the dashboard's 6-month trend | `getMonthlyCategoryTrend` in `src/lib/queries/monthly.ts` | Family enum is lowercase (`'coolant'`, `'washer_fluid'`, `'def'`) — current code has a known bug comparing to uppercase. |
| Add a migration | `db/migrations/NNN_*.sql` | Each file must have its own `BEGIN; ... COMMIT;`. Migrator does not wrap. |
| Wire a new chart | `src/components/charts/*` | Must be `"use client"`. Server pages pivot data and pass to client component. |
| Refresh MVs after a manual DB change | `SELECT u1d_ops.refresh_views();` | Order matters; the helper handles it. First-ever refresh of a new MV cannot use CONCURRENTLY. |

## Authentication (PR 003A)

The admin surface (/admin/*, /api/admin/*) is gated by NextAuth v5 with a Google OAuth provider. Sign-in succeeds only if the Google email matches a row in `u1d_ops.users` with `is_active = TRUE` and `role IN ('viewer','admin')`.

### Config split (why two files)

- **`src/auth.config.ts`** — edge-safe. Only Google provider, pages config, and the `authorized` callback (which checks JWT presence + URL pattern). Imported by `middleware.ts`.
- **`src/auth.ts`** — full config. Adds `signIn` (DB allowlist check via `lib/auth/allowlist.ts`), `jwt` (enrich token with role/isAdmin), and `session` (surface those on the client). Used by `/api/auth/[...nextauth]/route.ts`.

Never import `pg`-dependent code into `auth.config.ts` — middleware runs on the Edge runtime where `pg` does not exist. The allowlist check lives in `auth.ts` which runs on Node.

### Route protection matrix

| Path pattern | Middleware? | Behaviour |
|---|---|---|
| `/`, `/production`, `/reconciliation` | no | always public |
| `/login`, `/api/auth/*` | no | always public |
| `/admin/*` | yes | unauthenticated → 302 to `/login?callbackUrl=...` |
| `/api/admin/*` | yes | unauthenticated → 401 JSON |

### Session model

JWT strategy (no DB session store). The token carries `email`, `role`, `isAdmin`. `Session.user` is augmented in `src/types/next-auth.d.ts` so `session.user.isAdmin` typechecks.

### Common tasks for an AI agent

| Task | Where |
|---|---|
| Add a new protected route prefix | `src/auth.config.ts` ADMIN_PATH_PREFIXES + `src/middleware.ts` matcher |
| Add a new public route | `src/auth.config.ts` PUBLIC_PATHS |
| Tighten role checks (admin-only vs viewer-OK) | Page-level: read `session.user.role` from `auth()` and `redirect()` on mismatch |
| Force everyone to re-sign-in | Rotate `NEXTAUTH_SECRET` |

### What NOT to do

- **Do not** put `pg` imports or any Node-only code in `auth.config.ts`. Edge runtime will refuse to compile.
- **Do not** do allowlist re-checks on every request — that defeats the JWT optimization. The signIn callback is the gate.
- **Do not** silently fall back to "allow all" if the DB query fails. The current behavior denies sign-in on any throw, which is the safe default.
